
## 1. 文档目的

本文档记录 `prd_skill` 中已经拍板的关键技术与工程决策。

用途：

- 避免重复讨论已决定事项
- 帮助新成员和 AI 快速理解“为什么这么做”
- 记录当前方案的边界与代价

---

## D-001 当前采用 FastAPI + Pydantic v2 作为服务基础

- 状态：已确定
- 日期：`[待确认]`

### 决策
项目当前基于 FastAPI 和 Pydantic v2 构建 API 服务与 schema 边界。

### 原因
- 适合快速构建 API 服务
- 与当前项目规模匹配
- schema 定义与校验表达清晰
- 便于维持请求 / 响应边界稳定

### 影响
- 路由、schema、异常处理都围绕 FastAPI / Pydantic 体系组织
- 后续扩展应优先保持与当前结构兼容

---

## D-002 当前保持根目录平铺模块结构

- 状态：已确定
- 日期：`[待确认]`

### 决策
当前仓库继续采用根目录平铺方式组织主要模块。

### 原因
- 当前项目体量较小
- 核心代码文件数量仍可控
- 过早 package 化会增加不必要复杂度

### 影响
- 新增文件时应控制职责边界，避免继续膨胀
- 当模块数量和复杂度明显上升时，需要重新评估结构调整

---

## D-003 当前以分层清晰和边界稳定优先，而不是快速堆功能

- 状态：已确定
- 日期：`[待确认]`

### 决策
项目当前优先保持分层、抽象和可替换性，不以增加路由数量或快速堆叠功能为目标。

### 原因
- 项目未来需要扩展 provider、session store、执行链路
- 若早期边界混乱，后续扩展成本会快速上升

### 影响
- 路由层保持轻量
- provider 细节封装
- session store 抽象保持稳定
- schema 统一集中管理

---

## D-004 当前 session store 先使用内存实现

- 状态：已确定
- 日期：`[待确认]`

### 决策
当前使用 `InMemorySessionStore` 作为 session 存储实现。

### 原因
- 便于快速验证主链路
- 降低初期依赖复杂度
- 当前阶段不急于引入外部存储

### 影响
- session 不支持跨进程 / 跨重启保持
- 后续需要评估 Redis 等持久化方案
